# Yoga CRM Setup

This mini CRM is a static frontend that talks directly to Supabase for contacts, branches, and programs. It also includes a separate Programs tab inside the same CRM UI.

## Files

- `index.html` - single-page dashboard UI
- `styles.css` - responsive admin-style layout
- `app.js` - Supabase connection, contact CRUD, branch management, shared program API, filters, and follow-up actions
- `programs.js` - tab switching and program scheduling/linking UI with local cache + Supabase sync

## 1. Create the Supabase tables

Run this SQL in the Supabase SQL editor:

```sql
create table if not exists public.branches (
  id bigint generated by default as identity primary key,
  name text not null unique,
  created_at timestamptz not null default now()
);

create table if not exists public.contacts (
  id bigint generated by default as identity primary key,
  branch_id bigint references public.branches(id) on delete restrict,
  name text not null,
  phone text not null,
  location text,
  status text not null check (status in ('lead', 'trial', 'joined', 'completed', 'dropped', 'dried')),
  student_type text not null check (student_type in ('regular_sessions', 'hatha_yoga')) default 'regular_sessions',
  source text not null check (source in ('call', 'form', 'referral')),
  last_contacted date,
  next_followup date,
  notes text,
  created_at timestamptz not null default now()
);

create table if not exists public.programs (
  id text primary key,
  branch_id bigint not null references public.branches(id) on delete restrict,
  name text not null,
  scheduled_for date,
  notes text,
  whatsapp_template text,
  contact_ids jsonb not null default '[]'::jsonb,
  created_at timestamptz not null default now()
);
```

## 2. Allow the static site to use the table

Because this app has no authentication, the frontend uses the anon key directly. That means you must allow anon access to all three tables.

Run this SQL:

```sql
alter table public.branches enable row level security;
alter table public.contacts enable row level security;
alter table public.programs enable row level security;

create policy "anon can read branches"
on public.branches
for select
to anon
```

## 3. Add your Supabase URL and anon key

Open `yoga-crm/app.js` and replace:

```js
const SUPABASE_URL = "YOUR_SUPABASE_URL";
const SUPABASE_ANON_KEY = "YOUR_SUPABASE_ANON_KEY";
```

With your project values from:

- Supabase Dashboard -> Project Settings -> API -> Project URL
- Supabase Dashboard -> Project Settings -> API -> Project API Keys -> `anon` / `publishable`

## 4. Run or host it

This is hosting-ready as a static frontend. You can:

- open `index.html` directly in a browser, or
- deploy the `yoga-crm` folder to Netlify, Vercel static hosting, GitHub Pages, or any static host

## Included features

- Add, edit, delete contacts
- Add and delete branches yourself from the app
- Master branch filter at the top
- Filter by status
- Filter by student type
- Separate contact location field
- Multiple phone numbers per contact in the same phone field
- Mark follow-up done
- Copy a WhatsApp-style follow-up message
- Highlight follow-ups due today or overdue
- Separate `Contacts` and `Programs` tabs in the same CRM
- Programs tab for scheduling programs and linking contacts
- Programs stored in Supabase with browser localStorage used only as cache

## Programs Note

Programs now use Supabase as the source of truth.

That means:

- contacts, branches, and programs all come from Supabase
- browser localStorage is only used as a cache for faster reloads and fallback rendering
- if old browser-only programs exist in localStorage, the app will try to migrate missing ones into Supabase on load
